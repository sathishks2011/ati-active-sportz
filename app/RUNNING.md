# Active Sportz — local dev guide

This is the end-to-end guide for getting the M0 prototype running on your machine — prerequisites, project layout, day-to-day workflow, and common gotchas. Written assuming you're new to React Native; experienced RN folks can skim.

---

## 1. What you're running

The app at this milestone is **two throwaway UI prototypes**:

- **Setup screen** — three variants (A Free-draw, B Guided wizard, C Adjustable preset) for defining the Court ROI.
- **Recording screen** — three variants (A Driver dashboard, B Smart-film chip, C Heads-up overlay) for the live Session UI.

Each prototype has its own floating bottom pill that cycles A → B → C. A top pill flips between the two prototypes. A dev-only overlay drives mock Session State + motion score so you can see how each variant reacts.

Nothing is wired to real recording, real detection, or real persistence. That's M1 onward. This milestone is purely about picking design directions.

## 2. One-time machine setup (prerequisites)

You should only need to do this once per Mac. **You've already done these for this checkout** — they're listed for future machines or collaborators.

### 2.1 Node.js (≥ 22.11)

The project's `package.json` pins `"node": ">= 22.11.0"`. Recommended: install via [Volta](https://volta.sh) or [nvm](https://github.com/nvm-sh/nvm) so version switching is painless.

```sh
node -v   # should print v22.x or higher
```

### 2.2 Xcode (full version, not Command Line Tools)

Install **Xcode** from the Mac App Store. After install, make sure macOS uses Xcode's developer dir, not the bare Command Line Tools:

```sh
sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
sudo xcodebuild -runFirstLaunch
```

Verify:

```sh
xcode-select -p
# /Applications/Xcode.app/Contents/Developer

xcodebuild -version
# Xcode 26.5 (or newer)
```

### 2.3 iOS platform & simulator runtime

Xcode ships without the iOS SDK and simulator runtimes — those download on first use. Fetch them once:

```sh
xcodebuild -downloadPlatform iOS
```

That pulls ~5–10 GB and takes 15–30 minutes. After it finishes, check:

```sh
xcrun simctl list runtimes
# == Runtimes ==
# iOS 26.5 (...) - com.apple.CoreSimulator.SimRuntime.iOS-26-5
```

### 2.4 CocoaPods

CocoaPods is the iOS native-dependency manager that React Native uses.

```sh
sudo gem install cocoapods
pod --version   # should print 1.16+
```

### 2.5 Watchman

Used by Metro (RN's JS bundler) to watch files efficiently.

```sh
brew install watchman
watchman --version
```

### 2.6 (Recommended) direnv

The project uses an `.envrc` file to pin some env vars that pod install needs. `direnv` loads them automatically whenever you `cd` into the project.

```sh
brew install direnv
# Then add the appropriate hook to your shell rc:
#   bash:  echo 'eval "$(direnv hook bash)"' >> ~/.bashrc
#   zsh:   echo 'eval "$(direnv hook zsh)"' >> ~/.zshrc
# Restart your shell, then:
cd app && direnv allow
```

Without direnv you can `source .envrc` manually before running `pod install`.

## 4. React Native template choices (FYI)

Two facts you'll want to know when reading commands or future docs:

### 4.1 We use the Community CLI, not Expo

```sh
# This is the command that scaffolded the project:
npx @react-native-community/cli@latest init ActiveSportz --directory app --install-pods false
```

- **Community CLI** ("bare workflow"): gives you direct access to `ios/` and `android/` native projects, custom native modules just work, you call `xcodebuild` and `pod install` yourself.
- **Expo**: managed workflow with extra tooling. Easier for pure JS apps, but VisionCamera V5 + Frame Processors and our M3+ ML pipeline want bare access without Expo's `prebuild` dance.

Per [ADR-0008](../docs/adr/0008-tech-stack-rn-visioncamera.md), bare RN + VisionCamera V5 is the deliberate choice.

### 4.2 RN version & key libraries

- React Native `0.85.3` + React `19.x` + TypeScript `5.8.x`
- `react-native-vision-camera@^5` — camera capture + Frame Processors (Nitro Modules)
- `react-native-gesture-handler` + `react-native-reanimated` + `react-native-worklets` — drag gestures + worklet runtime
- `zustand` + `react-native-mmkv` — in-memory state + persistent KV
- `react-native-nitro-modules` + `react-native-nitro-image` — transitive Nitro deps that VisionCamera/MMKV pull in (declared as direct deps so RN autolinking sees them)

## 5. Per-checkout setup

You did this once already. You'll do it again if you ever delete `node_modules/`, `ios/Pods/`, or clone fresh on a new machine.

> **Before pod install: load `.envrc`.** The repo path contains a space
> (`My Workspace`), which trips RN 0.85's prebuilt-tarball downloader. The
> `.envrc` sets `RCT_USE_PREBUILT_RNCORE=0` and `RCT_USE_RN_DEP=0` to force a
> source build, which sidesteps the bug. Two ways to load it:
>
> - **One-shot (every shell):**
>   ```sh
>   source .envrc
>   ```
> - **Automatic via direnv (recommended):** install once (see §2.6), then
>   `direnv allow` from `app/` and the file loads on every `cd` into the
>   directory.
>
> If you forget, `pod install` fails with:
> ```
> [ReactNativeCore] Failed to download release tarball: bad component(expected absolute path component): ...
> [!] The `React-Core-prebuilt` pod failed to validate due to 1 error:
>     - ERROR | attributes: Missing required attribute `source`.
> ```
> The fix is always: `source .envrc && bundle exec pod install --project-directory=ios`.

```sh
cd app

# 1. Install JS deps
npm install

# 2. Install the Ruby gems that drive CocoaPods (uses ./Gemfile)
bundle install

# 3. Load env vars (skip this line if direnv is set up — it loads automatically)
source .envrc

# 4. Install iOS native pods
bundle exec pod install --project-directory=ios
```

After step 4 you should see:

```
Pod installation complete! There are 88 dependencies from the Podfile and 88 total pods installed.
```

The `88` may drift slightly as deps update. The important thing is no `[!]` error block at the bottom.

## 6. Running the app (simulator)

This is the everyday flow.

```sh
cd app

# Terminal 1 — start Metro (the JS bundler/dev server)
npm start
```

Leave that running. In a second terminal:

```sh
cd app

# Terminal 2 — build + boot the simulator + install + launch.
# This is the "one shot" command:
npm run ios -- --simulator "iPhone 17 Pro"
```

Behind the scenes that runs:

1. `bundle exec pod install` (skipped if up to date)
2. `xcodebuild` against the iOS Simulator destination
3. `xcrun simctl boot` the chosen simulator
4. `xcrun simctl install` the built `.app`
5. `xcrun simctl launch` the app, which connects to Metro at `localhost:8081` to fetch the JS bundle

If you'd rather drive each step manually (useful when debugging):

```sh
# Build for the simulator slice
xcodebuild -workspace ios/ActiveSportz.xcworkspace \
           -scheme ActiveSportz \
           -configuration Debug \
           -destination "platform=iOS Simulator,name=iPhone 17 Pro,OS=26.5" \
           -derivedDataPath ios/build/derivedData \
           build

# Boot + install + launch
xcrun simctl boot "iPhone 17 Pro" || true
open -a Simulator
xcrun simctl install "iPhone 17 Pro" ios/build/derivedData/Build/Products/Debug-iphonesimulator/ActiveSportz.app
xcrun simctl launch "iPhone 17 Pro" org.reactjs.native.example.ActiveSportz
```

### What you should see on the simulator

- **Camera backdrop** shows a dark fallback with "No camera available" — the simulator has no back camera. That's expected.
- **Top pill**: `Setup` ↔ `Recording`.
- **Bottom pill**: `‹ A — Free-draw ›` etc., cycles A → B → C.
- **(Recording prototype only)** Top-left compact horizontal panel:
  `state Capturing [step ›] | motion 0.50 [−] [+]`

Variant UI renders correctly even without a real camera feed — you can judge layout and density. To judge *how the chrome sits on top of live footage*, you'll want a physical iPhone (next section).

## 7. Running on a physical iPhone

The simulator is fine for layout, but the prototype's whole point is "how does each variant feel over a live camera feed?" — which needs a real device.

### 7.1 First-time device pairing

1. Plug your iPhone into the Mac via USB.
2. On the iPhone: trust the Mac when prompted.
3. Open Xcode → **Settings → Accounts** → sign in with your Apple ID (free Apple ID works for local dev; you don't need a paid Developer account for sideloading).
4. Open `app/ios/ActiveSportz.xcworkspace` in Xcode.
5. In the project navigator, click the **ActiveSportz** target → **Signing & Capabilities** → set **Team** to your Apple ID (free accounts show as "Personal Team"). Xcode will auto-generate a provisioning profile.
6. At the top of Xcode, click the **destination picker** and select your iPhone. Hit **⌘R** — the first run prompts Xcode to register the device with your team.
7. On the iPhone, the first launch shows "Untrusted Developer." Go to **Settings → General → VPN & Device Management → [your Apple ID] → Trust**. Then re-launch from the Home Screen or hit ⌘R again.

#### Bundle identifier — must be unique per developer

The RN template ships with `org.reactjs.native.example.ActiveSportz`, which Apple's free-tier signing rejects because every freshly-scaffolded RN project uses it.

This repo's bundle ID has been changed in `ios/ActiveSportz.xcodeproj/project.pbxproj` to `dev.sathishksoman.activesportz`. If you fork or use a different Apple ID, change both occurrences of `PRODUCT_BUNDLE_IDENTIFIER` (Debug + Release configs) to something unique to you:

```
PRODUCT_BUNDLE_IDENTIFIER = "dev.<your-handle>.activesportz";
```

After editing the file, close + reopen the Xcode workspace so it picks up the change.

#### Free Apple ID limitations

- Provisioning profile **expires after 7 days** — when the app stops launching on-device, just hit ⌘R from Xcode to re-sign and re-deploy.
- Max **3 registered devices** per Apple ID.
- Some entitlements (push, background audio, etc.) are unavailable on free accounts. None matter for this prototype; some will matter at M6 (interruption handling) if we wire local notifications — flag at that point.

### 7.2 On-device run

```sh
cd app
npm start   # Terminal 1, as before

# Terminal 2:
npm run ios -- --device "Your iPhone Name"
```

Or use Xcode: hit ⌘R with your device selected in the device picker.

The first run takes ~3 min while Xcode installs the build to the device. Subsequent runs are fast — Metro pushes JS over USB and the native bundle only rebuilds when you touch `ios/` or pod versions.

> If the first run prompts you on the iPhone to trust the developer profile, go to **Settings → General → VPN & Device Management → [your Apple ID] → Trust**.

## 8. Daily workflow

Once Metro is running, the loop is:

| Action | What happens |
|---|---|
| Save a `.ts` / `.tsx` file under `prototypes/` | Metro hot-reloads on the device/sim automatically — usually <1 second. |
| ⌘R in the simulator (or shake-device → "Reload" on iPhone) | Force a JS bundle reload. Useful when fast-refresh gets confused. |
| ⌘D in the simulator (or shake-device on iPhone) | Open the React Native dev menu — toggle perf overlay, element inspector, etc. |
| Touch anything in `ios/` (`.plist`, `Podfile`, native code) | `npm run ios` rebuilds the native bundle. ~30s. |
| Add a new npm package with a native module | `npm install …` → `bundle exec pod install --project-directory=ios` → rebuild. |

### Hot reload is JS-only

Metro fast-refresh **only** picks up JS/TS changes. If you change anything under `ios/` or update a `*.podspec`, you need a full rebuild.

   - The winning variant key (`A`, `B`, or `C`).

## 11. Common gotchas

### "Unable to find a destination matching..."
Your iOS platform isn't installed. Run `xcodebuild -downloadPlatform iOS`.

### "Failed to download release tarball: bad component" / "ERROR | attributes: Missing required attribute `source`" during pod install
The `.envrc` env vars aren't loaded in the shell. Fix:
```sh
source .envrc
bundle exec pod install --project-directory=ios
```
Long-term: install direnv (§2.6) and `direnv allow` so it loads automatically every `cd` into `app/`.

### "Unable to find a specification for X depended upon by Y"
A Nitro-modules-style native package isn't in `package.json`. Common culprits: `react-native-worklets`, `react-native-nitro-modules`, `react-native-nitro-image`. `npm install <pkg>` then re-run pod install.

### Metro shows "Could not connect to development server"
Metro isn't running, or the device can't reach `localhost:8081`. Make sure Terminal 1's `npm start` is still alive. On a physical iPhone over USB, Metro auto-tunnels — if not, plug into a Mac on the same network and shake → Configure Bundler Location.

### "EADDRINUSE: address already in use :::8081" when starting Metro
A previous Metro process is still holding port 8081. Kill it and retry:
```sh
lsof -ti:8081 | xargs kill -9
npm start
```

### "npm error enoent Could not read package.json"
You ran `npm` from the wrong directory. `package.json` is inside `app/`, not the project root. Always run JS/RN commands from `app/`:
```sh
cd "/Users/sathishksoman/Dev/My Workspace/Work/Projects/ATI-Active-Sportz/app"
```

### "Unable to find a device named ..." from `--simulator`
Simulator names are case-sensitive and must match `xcrun simctl list devices` exactly. `iphone 17 Pro` ≠ `iPhone 17 Pro`. Always copy the name from:
```sh
xcrun simctl list devices available | grep iPhone
```

### "Communication with Apple failed: Your team has no devices..." in Xcode signing
Two distinct problems folded into one message:
1. **Bundle identifier collision.** The default `org.reactjs.native.example.ActiveSportz` can't be re-signed under a free Apple ID. Change `PRODUCT_BUNDLE_IDENTIFIER` in `ios/ActiveSportz.xcodeproj/project.pbxproj` to something unique (e.g. `dev.<your-handle>.activesportz`), close + reopen the workspace.
2. **Device not yet registered to the team.** Pick your iPhone in the destination picker and hit ⌘R — Xcode prompts to register the device on the first run. The warning in Signing & Capabilities clears after that first run; you don't need to fix it from the pane itself.

See §7.1 for the full first-time pairing flow.

### "codesign wants to access key in your keychain" — what password?
The dialog wants your **Mac login password** (the one you type to log into macOS, same as `sudo`) — *not* your Apple ID password. Type it, then click **Always Allow** (not just "Allow") so codesign is added to the keychain item's ACL and the prompt doesn't recur on every build.

### "iOS 26.5 is not installed"
Same as the first gotcha — `xcodebuild -downloadPlatform iOS` and wait.

### Build succeeds but the app crashes on launch with "Worklets not installed"
The `react-native-worklets/plugin` Babel plugin isn't loaded. Check `babel.config.js` includes:

```js
plugins: ['react-native-worklets/plugin'],
```

### "xcode-select: error: tool 'xcodebuild' requires Xcode..."
`xcode-select` is pointing at Command Line Tools. Re-run:
```sh
sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
```

## 12. Where to look in code

- **Want to change how the bottom pill looks?** `prototypes/shared/PrototypeSwitcher.tsx`
- **Want to add/remove a mock control?** `prototypes/shared/MockControls.tsx`
- **Want to tweak a variant's chrome?** The corresponding `VariantA/B/C.tsx` under `prototypes/setup-screen/` or `prototypes/recording-screen/`.
- **Want to change permissions or app metadata?** `ios/ActiveSportz/Info.plist`.
- **Want to change which prototype mounts first?** Initial value in `prototypes/shared/MockStore.ts` (`activePrototype: 'setup'`).
