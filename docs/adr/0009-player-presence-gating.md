# Active Segment open requires player presence, not just motion

**Status:** Accepted (2026-05-21).

This decision is gated by a per-Session **Detection Mode** chosen by the user before tapping Auto Record (decisions-log: "Detection Mode names"). Three Modes are defined:
- **`'motion'` (UI: "Smart")** — the existing pipeline, unchanged in its open contract: an Active Segment opens on motion-in-Court-ROI sustained above `START_THRESHOLD` for `OPEN_HOLD_MS`. This Mode does not load the person-detector model.
- **`'players'` (UI: "Enhanced")** — additionally requires a low-rate on-device person detector to confirm at least `MIN_PLAYERS_IN_ROI` people inside the Court ROI at the open-confirmation tick.
- **`'continuous'` (UI: "Continuous")** — added 2026-05-21. Bypasses detection entirely. No Court ROI is captured at Setup, no Active Segments are emitted, no splice runs on Stop. The Master Recording is saved directly to Photos as the user-facing video. Useful for non-court captures (general recording) and for validating the recorder path in isolation from the detection pipeline. Configured the same way as the other two — picked in Settings or via the Setup Step 2 segmented control. Calibrating phase is skipped automatically (no baseline to learn).

When Mode is `'players'`, an Active Segment opens only when **both** signals agree: (a) motion-in-Court-ROI sustained above `START_THRESHOLD` for `OPEN_HOLD_MS`, **and** (b) the person detector reports at least `MIN_PLAYERS_IN_ROI` people inside the Court ROI for that tick. The segmenter's input contract gains a new path `onTick({motionScore, personCount?, atMs})` alongside the existing `onScore` — when `personCount` is `undefined` (Mode = `'motion'`), behaviour is identical to today's `onScore`. The closing path is **not** changed in either Mode — once a Segment is open, the existing trailing-hold (8s under `END_THRESHOLD`) and motion-only criterion still govern close, so brief mid-rally lulls (between points, server walking back) do not flicker the Segment.

Both Modes adopt the quadrilateral Court ROI introduced in ADR-0010 — "inside the Court ROI" means inside the user-tapped polygon, not a screen-aligned rectangle. This keeps the A/B comparison between Modes honest: exactly one variable differs (the player-count gate), not two.

This supersedes the relevant clause of ADR-0007 that describes Active Segment emission as motion-only ("inference pipeline … emits Active Segment timestamps … as motion-in-Court-ROI crosses thresholds"). The continuous-Master architecture itself is unchanged: Active Segments remain metadata pointing into one continuous Master Recording.

Reason: the MVP detector is a pure luminance-diff signal sampled inside the Court ROI. In testing, the "any motion → Capture" trigger conflates real gameplay with non-gameplay activity inside the same ROI — a coach pacing the sideline (if drawn inside the ROI), a single player retrieving a stray ball, parents crossing the court between sets, a referee setting up the net. The leading-hold decision (decisions-log) kills the *transient* class of false positives but does nothing about *sustained* non-game motion. The category of bug we're fixing is not "pixels changed" but "things-changing-on-camera ≠ game-is-happening." A semantic signal is required to tell the two apart. `react-native-fast-tflite` with the CoreML delegate (already adopted in ADR-0008) is the established path for on-device inference in this stack, so this is an extension of the existing pipeline rather than a new dependency.

Considered alternatives:
- **Status quo + leading-hold only** (shipped): cheap, kills transient walk-throughs, but does not address sustained single-person motion inside the ROI. Insufficient on its own once we tested in real gyms.
- **Tighten the Court ROI (polygon, not rectangle; user-drawn court boundary)**: improves rejection of sideline traffic but does not address motion *inside the court itself* that isn't gameplay (coach demonstration, line-judge crossing). Worth doing independently, but not a substitute.
- **Require motion in multiple sub-regions of the ROI (e.g., near-court AND far-court must both be active)**: pure-geometry, no ML, partially solves the "one person on the court" case. The quadrilateral ROI from ADR-0010 makes a future both-sides-of-net check tractable (split the homography-mapped y-axis at 0.5) — but we're not implementing it in this round; the `'players'` Mode's `MIN_PLAYERS_IN_ROI` gate is expected to do enough.
- **Pose / activity-recognition models (MoveNet pose, action classifiers)**: stronger signal, but materially heavier than person detection and a much larger tuning surface (which poses are "playing"?). Phase 2.
- **Ball detection** (track the volleyball itself): dropped from scope entirely. Volleyball-at-gym-distance is a hard CV problem (small, fast, motion-blurred, leaves frame on serves) and the off-the-shelf "sports ball" class on COCO-trained detectors has poor recall at this scale. If field data from `'players'` Mode ever shows player-presence is insufficient, ball detection becomes the trigger for a future ADR — not a tracked roadmap item.
- **Audio cues** (whistle, ball impact, crowd reaction): explicitly out of scope per ADR-0001 (no audio in MVP).
- **Shadow mode** (run both Modes simultaneously on every Session and produce two Session Recordings per Master): considered, rejected. Pays V2 inference cost on every Session even when only V1 is being evaluated; doubles splice time at Stop; complicates persistence with a per-segment pipeline-version tag. Replaced by the per-Session user-selectable Mode toggle, which trades the "same-take comparison" property for a much simpler runtime and persistence model.

Consequences:
- **Segmenter API gains `onTick`.** `onTick({motionScore, personCount?, atMs})` is added alongside the existing `onScore(score, atMs)`. When `personCount` is `undefined` (Mode = `'motion'`), behaviour is identical to today's `onScore`. When `personCount` is present (Mode = `'players'`), the open-confirmation tick requires both the existing motion-hold *and* `personCount >= MIN_PLAYERS_IN_ROI`. The host (RecordingScreen) drives the appropriate path based on the per-Session Mode, with the person-count refreshed at a lower cadence than motion (likely 2 Hz vs 10 Hz). `config.ts` gains `MIN_PLAYERS_IN_ROI` (initial guess: 3 — enough to reject 1–2-person non-game activity, low enough to tolerate per-frame detector misses during real gameplay).
- **Per-Session Mode persisted.** `sessions.detection_mode` (TEXT, `'motion'` or `'players'`) records which Mode produced each Session. The Library shows a badge per Session entry so the user can tell which pipeline produced which recording when comparing field-test takes.
- **New inference pass in the Frame Processor pipeline.** The motion worklet stays as-is (Y-plane diff). A second pass runs the person detector on a downscaled RGB plane every ~500ms. CoreML delegate; model choice TBD between MoveNet Lightning, EfficientDet-Lite0, and SSDLite-MobileDetV2 — to be decided by an on-device latency + accuracy bake-off, not on paper.
- **New asset footprint.** The chosen TFLite model adds ~4–10 MB to the bundle. Acceptable on iOS.
- **Warm-up phase implications (ADR-0006).** During Warm-up the person detector also runs, but its output is not used to gate Segment-open (which is suppressed during Warm-up anyway). Warm-up's existing 15s window is sufficient for the detector to spin up; no additional warmup logic needed.
- **Close path unchanged.** A Segment, once open, closes purely on the existing motion-based trailing hold. Rationale: players don't physically leave the court between points, so the person signal would near-always vote "still active" and add no information at close time — but it would add a new failure mode (one-frame detector miss during a serve causes premature close). Keeping close motion-only is the simpler, more robust default; revisit only if real-data shows premature closes are not the dominant failure.
- **Thermal / battery cost.** The motion worklet currently dominates the inference budget. Adding a 2 Hz person-detector pass on the Neural Engine via CoreML should be additive but small; needs measurement in M7 against the 90-min target. If it pushes thermal envelope, the first lever is dropping detector cadence to 1 Hz (still much faster than the 1.5s `OPEN_HOLD_MS` window, so it does not change behaviour materially).
- **Trigger-contract terminology.** CONTEXT.md's "Active Segment" glossary entry currently describes detection as motion-based. The entry needs updating to reflect that detection is motion ∧ player-presence. No new glossary term needed.
- **Tuning surface expands.** Pre-ADR there was one dial that mattered for open (`START_THRESHOLD`) and two for close (`END_THRESHOLD`, `TRAILING_HOLD_MS`). Post-ADR adds `MIN_PLAYERS_IN_ROI`, detector cadence, and model choice. Each is a separate failure mode in the field.

Hard to reverse (the segmenter input contract, the worklet pipeline, and the bundle assets all change; reverting means a measurable code + asset diff); surprising-without-context (a future engineer reading "why does the camera path run a person detector?" deserves to find this ADR, especially given the original architecture was deliberately ML-free at detection time); the result of a real trade-off (CPU/thermal + tuning-surface complexity vs. trigger precision, with several credible non-ML alternatives evaluated above).

Outstanding items, to be resolved during implementation (do not block Accept):
- **Model choice** — on-device bake-off across MoveNet MultiPose Lightning, EfficientDet-Lite0, SSDLite-MobileDetV2, and YOLOv8n. Judged on per-frame latency via the `fast-tflite` CoreML delegate (target < 200 ms so 2 Hz is comfortable), recall of "people inside a manually-traced court polygon" on three-gym test clips, and model size < 10 MB. The chosen `.tflite` drops into `src/detection/personFrameProcessor.ts` behind a `TODO(bake-off)` shim that's already in place.
- **`MIN_PLAYERS_IN_ROI` initial value** — start at 3, retune from real-match field-test logs once available.
- **Both-sides-of-net check** — not implemented in this round. The quadrilateral Court ROI from ADR-0010 enables it via homography (split mapped-y at 0.5); revisit only if `MIN_PLAYERS_IN_ROI = 3` proves insufficient.
- **Person-detector failure behaviour** — when the model fails to load or crashes at runtime, fall back to `'motion'` Mode for the remainder of the Session and surface a one-time in-app warning. This degrades to today's known-shippable behaviour rather than blocking the Session entirely.
