# Continuous Master Recording + metadata splice

During a Session, the camera captures a single continuous video file — the **Master Recording** — without ever pausing or restarting the hardware encoder. In parallel, an inference pipeline (Frame Processor + TFLite model where applicable, or plain frame differencing) emits **Active Segment** timestamps to a metadata store (JSON file or local DB row) as motion-in-Court-ROI crosses thresholds. At Stop time, a splice step (FFmpeg) reads the Master Recording and the Active Segment metadata and produces the user-facing **Session Recording** by joining the active spans. The Master Recording is retained as a first-class persisted artifact with user-controlled retention. This supersedes ADR-0003 and ADR-0005.

Reason: starting and stopping the hardware video encoder per-segment (as the original Model Y architecture implied) introduces three serious problems even in pure native code — encoder spin-up latency (missed start of plays), risk of corrupted segment files at boundaries, and elevated thermal/battery cost from constant session resets. A single continuous encode session is exponentially more reliable and efficient. It also makes the pre-roll concept obsolete: every Active Segment automatically has full lead-in footage because the Master always contains it. The trade-off — storage cost of the Master file (~3–6 GB per 90 min at 1080p hardware H.264) — is acceptable on modern phones, especially when we expose user-controlled retention rather than enforcing eviction.

Consequences:
- **Active Segments are metadata, not files.** Each Active Segment is an entry with start/end timestamps and detection-signal scores, pointing into the Master Recording. The conceptual entity from ADR-0003 survives; only the representation changes.
- **Durability granularity is the Master Recording + incremental metadata writes.** The metadata store is appended on each Segment open/close. On crash, the partial Master file + the (last-successful) metadata state defines what's recoverable. Silent recovery on next launch (as in ADR-0005's principle) still applies — the app finalizes whatever was captured and produces a Session Recording from the surviving metadata.
- **Pre-roll concept is obsolete.** The 5-second pre-roll from the original tuning is irrelevant under this architecture; the Master always has the lead-in. The trailing-hold (8s) remains relevant as a hysteresis parameter on the *metadata* — it controls when an Active Segment is "closed" in metadata, not when a file stops writing.
- **Master Recording retention is user-controlled, not automatic.** Defaults: Master is kept indefinitely until the user invokes an in-app "Clean up" action. We do not auto-delete on Session-end (that throws away the most valuable recovery affordance — re-splicing with corrected metadata if the detector was wrong). We do not auto-delete on age (silent data loss is a worse failure than a full-disk warning). The user gets a clear in-app view of disk used by Sessions and a per-Session "Delete Master Recording (keep Session Recording)" option, plus a global "Delete Sessions older than N days" sweep.
- **Splice time at Stop is not instantaneous.** FFmpeg `-c copy` is fast but only cuts cleanly at keyframe boundaries (~2-second granularity). For arbitrary timestamps we accept ~2s alignment slop or pay re-encode cost at segment boundaries. For a 90-minute Session with ~30 Active Segments, expect 30–90 seconds of post-Stop processing. The "Stopping" Session State must show progress; users are not surprised by a slow Stop.

This decision is hard to reverse (the entire capture/storage layout depends on it), genuinely surprising vs. the original `idea.md` plan (which implied per-clip recording), and the result of a real trade-off against the original Model Y approach (which had its own real merits, captured in the now-superseded ADR-0003).
